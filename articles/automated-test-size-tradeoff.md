Q. 「自動テストを書いてきてしばらく経つが、実行時間の増加に伴う問題 (CI 待ち時間) や偽陽性・偽陰性の問題が出てきた。どう対応していくのがよいか？」

IMO: **コードの変更に対する不安**と**実行時間**をコンパスにして、自動テストの肥大化に立ち向かう。(一ヶ月後には言っていることが変わるかもしれないので、今日のスナップショットとして聞いてもらえればと...)

## そもそも「よいテスト」とは

テストを書くべき理由はいろいろありますが、**テストに何を期待するか**は明確にしておく必要があります。私は以下のどちらかを満たすものだと考えています。

- **開発者が安心できる**
- **保守性にポジティブな影響を与える**

逆にこれらを満たさないテストは、価値が低い (あるいはマイナス) と言えるかもしれません。人間と AI の認知リソース、コンピュータの計算リソース、CI の実行時間を浪費し続けるからです。

この整理は lacolaco さんの登壇からインスパイアされたものです。

https://blog.lacolaco.net/posts/tskaigi-2026-deck
https://blog.lacolaco.net/posts/presentation-phpconfuk-testing-dom

### 開発者が安心できる

自動テストがあっても安心できなければ、結局こういう行動に走ります。

- 毎回手動でテストしている
- コードレビューでバグを見つけようとする

これでは自動化してもメリットの一部しか享受できません。理想は**テストが通っていれば自信をもってリリースできる**状態で、不安を感じるならそこに改善の余地があります。

### 保守性にポジティブな影響を与える

テストを書く過程で設計について深く考えることになります。例えば、「テストを複雑にしてまで関数にパラメータを追加するべきか」を考えると、高凝集・粗結合になりやすいとか。

- テストの書きにくさは何かが間違っている予兆
- テストが通っている (壊れたら気づける) からリファクタリングできる
- TDD でより良い設計をひらめく瞬間が増える
- テストがユースケースの表明になり、コードが自己ドキュメント化される

https://note.com/erukiti/n/nee6f1a911307

## テスト粒度のトレードオフ

テストの粒度は実行時間やいくつかの特性とトレードオフの関係にあります。例えば、実行時間を短縮するために小さい粒度のテストに傾倒するとオオカミ少年になりやすく、より多くのバグを見つけようと本番に近い環境ばかりでテストを実行すると実行時間が爆発します。両取りはできません。このトレードオフのつまみを調整して、**スウィートスポットを探り続ける**のが仕事だと思っています。

### テストサイズという共通言語

テストを実行するのに必要なリソースで機械的に Small / Medium / Large に分類します。

- Small: 単一プロセスに閉じる (ネットワーク・ファイルシステムへのアクセスを禁止)
- Medium: 単一マシンで完結する (ファイルシステム・ローカルホストへのアクセスを許可)
- Large: 複数マシンにまたがる (ローカルホスト外へのアクセスを許可)

https://abseil.io/resources/swe-book/html/ch11.html

https://speakerdeck.com/twada/automated-test-knowledge-from-savanna-202406-findy-dev-prod-con-edition?slide=33

サイズで切る理由は、実行速度と決定性という運用上の関心ごとに集中し、機械的にラベリングできるからです。具体例で対応づけると、こんな感じのイメージになります。

| サイズ | 例                                                                                                                                                                                                  |
| ------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Small  | Vitest だけで完結するテスト (テスティングトロフィーでいう結合テストの多くもここに入る)                                                                                                              |
| Medium | Vitest × [Testcontainers](https://testcontainers.com/) (本物の DB)、[LocalStack](https://www.localstack.cloud/) で外部サービスをエミュレート、Playwright でフロントエンドからバックエンドまでテスト |
| Large  | デプロイ済み環境に対する E2E テスト (認証込み)、AWS 環境に接続するテスト                                                                                                                            |

## テストサイズで再翻訳する

テストピラミッドやテスティングトロフィーの定義をテストサイズの分類に当てはめてみます。

![](/images/testing_concept_with_test_sizes.png)
_テストサイズ (Small / Medium / Large) に対応づけた図_

Small テストに寄せたテストピラミッドや、Small / Medium テストに寄せたテスティングトロフィーをつくれるため、プロジェクト内で語彙を揃えてテスト戦略を議論できます。このテクニックは AI とのコンテキスト共有でも有効だと考えています。

## テストは AI に書かせますよね？

AI にテストを書かせる時代だからこそ、テストサイズによる分類が効いてきます。Design Doc のテスト戦略が AI の出力を大きく左右するコンテキストになるからです。
改めて「よいテスト」の定義に立ち返ってテストサイズと関連付けると、

- **保守性にポジティブな影響を与えるのは Small テスト**
- **安心できるなら、(高速に実行できる) より小さいテストがよい**

といったことが言えそうです。

AI はテストを書くのが得意ですが、テスト観点や受け入れ基準 (AI にとって十分なコンテキスト) とテスト戦略 (出力に大きく影響) を与えるのは人間の仕事です。
前に書いた通り、価値のないテストが量産されてしまう前に「**〇〇 (特性や実行時間) を重視するため、どのテストサイズに比重を置くか**」というプロジェクトにおける判断を行えるのがベストだと思います。ただ、発初期では判断に必要な情報が揃っていない状況の方が多いと思うので、後述する運用で継続的に改善していきます。

## レイヤードアーキテクチャとの関係

レイヤードアーキテクチャはテストしやすいコードを強制するギプスにもなります。[クリーンアーキテクチャ (The Clean Architecture)](https://blog.cleancoder.com/uncle-bob/2012/08/13/the-clean-architecture.html) では、ユースケースが DB の具象に依存しないよう、ドメイン層にリポジトリのインターフェースを置いてそれに依存させると思います。

```ts:domain/spot/repository.ts
// リポジトリのインターフェースはドメイン層に置く
export type SpotRepository = {
  create: (spot: Spot) => ResultAsync<SpotId, DatabaseError>;
  findMany: () => ResultAsync<Spot[], DatabaseError>;
  // ...
};
```

これにより DIP (依存性逆転の原則) が強制され、ユースケースのテストは DI (依存性注入) でリポジトリをテスト用の実装に差し替えられるようになります。

```ts:application/usecase/spot/list.ts
// ユースケースはリポジトリのインターフェースにだけ依存する
export function ListSpotsUsecase({ spotRepository }: { spotRepository: SpotRepository }) {
  return {
    execute: () => spotRepository.findMany(),
  };
}
```

```ts:application/usecase/spot/list.test.ts
// テストではリポジトリをテスト用の実装に差し替える
it("スポットの一覧を返す", async () => {
  const spots = [createSpot(), createSpot()];
  // モックしたリポジトリを使う
  const spotRepositoryMock = createSpotRepository({
    findMany: vi.fn().mockReturnValue(okAsync(spots)),
    // ...
  });

  // DI でリポジトリを依存注入する
  const result = await ListSpotsUsecase({ spotRepository: spotRepositoryMock }).execute();

  expect(result._unsafeUnwrap()).toEqual(spots);
});
```

また、**レイヤーごとにテストで何を担保するか**を考え、それを設計に適用することが重要です。テストを拡充していくと、必ずレイヤーを跨いで重複する部分が出てくるのですが、同じバグしか検知できないテストが何個あっても結果は同じです。ベン図で表現するところの重なりを無くすことは不可能 (その必要もない) ですが、無駄な重なりが発生しにくいようなテスト戦略は心がけています。

![](/images/layered-tests.png =400x)
_レイヤーごとのテストが検証できる範囲_

:::message
テスト手法についても無駄な重複を排除します。例えば、VRT があるのに Testing Library で DOM の描画テストをしても仕方がありません (責務が重複しており、なおかつ包含関係にある)。
:::

## テストサイズを下げる手法

テストサイズを下げることで、環境の忠実性を引き換えに**より速いテスト**を得ることができます。ここでは主要なパターンを 2 つ紹介します。

### DB をモックする (Medium → Small)

先ほど見たユースケースのテストのように、リポジトリをモックすることで単一プロセス (Vitest のみ) で実行できる Small テスト になります。ただし、リポジトリをモックするということは SQL はテストされないということです。以下のように、リポジトリの Medium テストを別で書いていなければ SQL のバグを検知できません。

```ts:infrastructure/query-services/spot-summary.ts
// スポット一覧にレビュー件数・平均評価を添えて返す
findMany: (): ResultAsync<SpotSummary[], DatabaseError> => {
  return ResultAsync.fromPromise(
    db
      .select({
        id: spots.id,
        name: spots.name,
        reviewCount: count(reviews.id),
        averageRating: avg(reviews.rating),
      })
      .from(spots)
      // バグ: 外部結合 (leftJoin) にすべきところが内部結合 (innerJoin) になっている → レビューが 0 件のスポットが一覧から抜け落ちる
      .innerJoin(reviews, eq(reviews.spotId, spots.id))
      .where(
        notExists(db.select().from(archivedSpots).where(eq(archivedSpots.spotId, spots.id))),
      )
      .groupBy(spots.id),
    (error) => ({ kind: "DATABASE" as const, message: String(error) }),
  ),
},
```

### 外部サービスをモックする (Large → Medium)

AWS S3 への保存のような外部サービス連携は、実際の環境に繋ぐと Large テストになります。ここで LocalStack のようなクラウドサービスの実行環境をエミュレートできるツールを使うと、単一マシン上で動く Medium テストとして扱えます。ここでもトレードオフがあり、例えば外部サービスとの整合性などは検証できません。

```ts:infrastructure/gateways/s3-photo-storage.medium.test.ts
// LocalStack を Testcontainers で起動する
const s3 = new S3Client({
  endpoint: inject("s3Endpoint"),
  forcePathStyle: true,
  credentials: { accessKeyId: "test", secretAccessKey: "test" },
  region: "ap-northeast-1",
});
const storage = S3SpotPhotoStorage(s3, "spot-photos");

const result = await storage.save({ key: "photo-1.png", body, contentType: "image/png" });

expect(result.isOk()).toBe(true);
```

:::details 型検査に寄せると単体テストが減る
ドメインの仕様を型で表現すると、そもそも単体テストを書かなくて良いケースが増えます。例えば、[Always-Valid Domain Model](https://enterprisecraftsmanship.com/posts/always-valid-domain-model) の考え方を徹底することで、型レベルでそれ以外の値が入らないことを保証してくれます。そのため、想定された入出力に対してのみテストを書けばよくなります。

```ts
// エンティティの状態を型で表現
type ArchivedSpot = Spot &
  Readonly<{
    archivedAt: Date;
  }>;

// (DB からの復元などを除いて) アーカイブ状態のスポットは必ずこの関数から生成される
// ※ この例は Brand 型を使用していないので、型キャストで嘘をつけます
function archiveSpot(spot: Spot): ArchivedSpot {
  return {
    ...spot,
    archivedAt: new Date(),
  };
}
```

また、サンプルコードでは [neverthrow](https://github.com/supermacro/neverthrow) を採用し、関数が返すエラーを種別ごとにカスタムエラー型へマッピングして返すように設計しています。そうすることで、関数シグネチャからテストすべきケースが読み取りやすくなり、どこで例外がスローされるか考えなくて済みます (関数シグネチャに現れないテストは不要)。

```ts
// 識別用のタグ (kind) を持ったカスタムエラー
type ValidationError = { kind: "VALIDATION"; message: string };

function Spot({
  description,
  id,
  latitude,
  longitude,
  name,
}: {
  description: string | null;
  id: SpotId;
  latitude: number;
  longitude: number;
  name: string;
  // ValidationError が返る可能性があることがシグネチャに現れる
}): Result<Spot, ValidationError> {
  return Coordinate({ latitude, longitude }).map((coordinate) => ({
    coordinate,
    description,
    id,
    name,
  }));
}
```

:::

## REST API の例

データベースと CRUD が中心のアプリケーションのバックエンドを REST API で実装するとします。テスト戦略の一例として、次のように考えてみました。

| レイヤー                  | サイズ | 網羅性                             | DB / 外部サービス / その他                                                                                    |
| ------------------------- | ------ | ---------------------------------- | ------------------------------------------------------------------------------------------------------------- |
| プレゼンテーション        | Medium | ステータスコードごと (正常・異常)  | 本物の DB / モック / フロントエンドサーバーは起動しない                                                       |
| ユースケース              | Small  | 正常・異常 (エラーコード) ごと     | DB と外部サービスはモック                                                                                     |
| ドメイン                  | Small  | 入出力の条件を網羅                 | 基本的に他のレイヤーに依存しない (同一レイヤー内では依存を許可)。ドメインサービスは DB と外部サービスをモック |
| リポジトリ                | Medium | 正常・異常ごと                     | TestContainers でコンテナ内で本物の DB を起動                                                                 |
| ゲートウェイ (アダプター) | Medium | 正常・異常ごと                     | LocalStack で外部サービスをエミュレート                                                                       |
| フロントエンドとの結合    | Medium | ユースケースごと・クリティカルパス | 本物の DB / モック / フロントエンドのユーザー操作から一気通貫でテスト                                         |

ドメイン層はクリーンアーキテクチャの最も内側に位置し、変わりにくい重要なロジックが多く含まれる部分です。ここは条件網羅で手厚くテストします。Small テストが中心になるため、実行時間にも大きな影響を与えないという判断です。

```ts:domain/spot/models/coordinate.test.ts
it.each([
  [-90, true], [90, true], [0, true],   // 境界値の下限・上限・範囲内
  [-90.1, false], [90.1, false],        // 境界値違反
])("latitude=%s → ok=%s", (latitude, ok) => {
  expect(Coordinate({ latitude, longitude: 0 }).isOk()).toBe(ok);
});
```

雑ですが、その他レイヤーの判断基準は以下の通りです。

- プレゼンテーション: API 視点で見た消費者 (フロントエンド) との接点なので、本物の DB を使って SQL まで通した分岐網羅でテストする。実際の外部サービスに繋ぐと Large テストになってしまうためモックし、フロントエンドまで含めるのは E2E の領分なのでここではやらない
- ユースケース: ドメインやインフラのコードはテストされている前提で、関数の入出力に関わる部分だけ分岐網羅でテストする
- リポジトリ: SQL までテストしたいので、本物の DB を使って関数の入出力に関わる部分だけ分岐網羅でテストする
- ゲートウェイ: 単一の部品で Large サイズのテストになるのは避けたいので、何らかのモックを使う。関数の入出力に関わる部分だけ分岐網羅でテストする
- フロントエンドとの結合: 正常系を中心としたユースケースとクリティカルパスに絞って、フロントエンドから一気通貫でテストする。外部サービス込みのテスト環境を用意するのは経験がないので知らない

## 全件テストはありえない

AI の台頭で力点は変わりましたが、今後も人間のチェックがなくなるとは思いません。テストはバグがないことの証明にはならないので、結局のところリリース可否の判断は人間が安心できるかで決まります。

- 設計段階で仕様が抜け落ちていないか
- 設計を正しく実装できているか
- それらを突き合わせて検証するテストがあるか。その検証方法は正しいか

## 運用

プロジェクトの状況に応じて、テストに対する考え方や期待値も変えていく必要があります。基本的に 1 度書かれたテストは消えにくく、コード量と比例して増えます。また、開発初期では安心を買えるサイズの大きなテストに比重を置くのも有効ですが、コードの成長に従ってどこかで実行時間の壁にぶち当たります。

冒頭のトレードオフの話に戻りますが、変化を捉えて改善するための仕組みが重要です。

> トレードオフのつまみを調整して、スウィートスポットを探り続ける

そのために、**テストカバレッジや実行時間を定期的に収集・分析**し、つまみを調整するアクションに繋げます。この辺は特に AI が得意な分野だと思うので、分析結果をもとに以下のシンプルな決定フローで判断します。

- **実行時間を買いたい** → 実行に時間がかかるテストを間引く / テストサイズを下げる / 実行タイミングを後ろにずらす / 部分的にテスト以外のもっと高速な方法 (静的チェックなど) で代替する
- **安心を買いたい** → 発生したバグから偽陰性の原因を分析してテストに落とし込む / カバレッジを上げる (不足しているテスト観点の追加やテスト戦略の見直し) / テストサイズを上げる / 実行タイミングを前に前に持ってくる / 部分的にもっと確実な方法 (手動テストなど) で代替する

また、テストの棚卸し (無駄なテストの削除、flaky テストの対策など) は何れにせよ有効なので、定期的な運用タスクに込むこむのが良いと思います。

## おわりに

書いたことをどれもできている訳ではないので頑張りたいです。
